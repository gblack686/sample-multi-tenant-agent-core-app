# SSE Plan: E2E Test for Stream Output, Schema Match, and Rendering

## Overview

- **Change Type**: Test improvement — new Playwright spec + enhance validate-uc
- **Layers Affected**: Frontend tests only (no backend changes)
- **Files Modified**: 1 new spec, 1 enhanced spec

## Problem

We have 3 separate test files that partially cover SSE:

| File | What it checks | Gaps |
|------|---------------|------|
| `validate-tool-cards-chevron.spec.ts` | SSE capture, tool_use:tool_result ratio, chevron presence | No schema validation, no rendering check |
| `validate-uc-simple-acquisition.spec.ts` | Response quality, keywords, session persistence | Weak tool card detection (selector guessing), no SSE capture |
| `validate-chat-v2.spec.ts` | CopilotKit AG-UI events | Targets `/chat-v2` (not primary), no SSE stream check |

**What's missing:**
1. **Schema match** — verify SSE event JSON matches `StreamEvent` type (has `type`, `agent_id`, `agent_name`, `timestamp` + correct payload field)
2. **tool_use→tool_result pairing** — verify every tool_use has a tool_result with matching name
3. **Frontend rendering** — verify tool cards render with correct label, status dot, and expandable result (not raw JSON leak)
4. **Unified UC test** — a single test that validates the full pipeline from SSE wire → parse → state → render

## Design

### Option A: Enhance `validate-uc-simple-acquisition.spec.ts` (recommended)

Add SSE capture + schema validation + rendering checks to the existing UC test. This gives us one comprehensive test instead of 3 partial ones.

### Option B: New standalone `validate-sse-pipeline.spec.ts`

Dedicated test that focuses purely on SSE integrity, schema match, and rendering. Can be run independently.

**Recommendation: Do both.** Option B as a focused regression test for SSE, Option A as the integrated UC test.

## Changes

### File 1: `client/tests/validate-sse-pipeline.spec.ts` (NEW)

Focused SSE pipeline test:

```
Test: "SSE pipeline: schema, pairing, and rendering"

1. Navigate to /chat
2. Intercept /api/invoke response → capture raw SSE lines
3. Send a query that triggers multiple tools (e.g., thresholds question)
4. Wait for response complete
5. Parse all SSE events and validate:

   SCHEMA CHECKS:
   - Every event has: type, agent_id, agent_name, timestamp
   - text events have: content (string)
   - tool_use events have: tool_use.name, tool_use.input (object)
   - tool_result events have: tool_result.name, tool_result.result (object)
   - complete events exist (exactly 1)
   - No events with empty/null type

   PAIRING CHECKS:
   - Count tool_use == count tool_result
   - For each tool_result.name, a matching tool_use.name exists
   - No tool_result with empty name (filter worked)

   RENDERING CHECKS:
   - For each tool_use, a tool card exists in DOM (.my-1.rounded-lg.border)
   - Each card has: icon, label, green status dot
   - Each card has: chevron (▾) character
   - Click chevron → expanded panel appears with content
   - Expanded content is NOT raw "{"type":"tool_result"..." (no SSE leak)
   - Expanded content has structured display (JSON block or markdown)

   ASSERTIONS:
   - expect(schemaErrors).toHaveLength(0)
   - expect(toolResultCount).toBe(toolUseCount)
   - expect(cardsWithChevron).toBe(totalCards)
   - expect(expandedContentIsFormatted).toBe(true)
```

### File 2: Enhance `validate-uc-simple-acquisition.spec.ts`

Add to existing Step 6 (tool cards):

```
6a. SSE CAPTURE (add response interceptor like chevron test)
    - Capture tool_use and tool_result events from /api/invoke
    - Assert tool_use count == tool_result count
    - Assert every tool_result has non-empty name

6b. TOOL CARD RENDERING (replace weak selector guessing)
    - Use .my-1.rounded-lg.border selector (from tool-use-display.tsx)
    - Check each card for chevron (▾)
    - Click first chevron to expand
    - Verify expanded content is present and formatted

6c. SCHEMA SPOT-CHECK
    - Verify at least 1 tool_use event has agent_id and timestamp
    - Verify at least 1 tool_result event has result object
```
